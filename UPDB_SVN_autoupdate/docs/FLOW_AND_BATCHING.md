# 輸入檔處理流程與「每專案分次設定」說明

## 輸入範例如何被解析

每行以 **Tab** 區隔第一欄與第二欄；第二欄起為工號與其他欄位（程式只取第二欄第一個子欄位作為工號）。

範例：

```
RL6483 Digital RT	R9901	柯佳佑	3518704	通訊網路事業群	 joeke@realtek.com
RL6665 Digital RT	R9901	柯佳佑	3518704	通訊網路事業群	 joeke@realtek.com
```

`parser.py` 規則：

- **第一欄（Tab 前）**：以空白拆成 token，第 1 個 = 專案名、第 2 個 = 群組名；若有 `SVN` 則 `add_svn=True`（目前程式已移除 SVN 功能，此欄位僅解析不使用）。
- **第二欄（Tab 後）**：第一個子欄位作為工號。

因此會得到兩筆 Record：

| 專案   | 群組    | 工號  |
|--------|---------|-------|
| RL6483 | Digital | R9901 |
| RL6665 | Digital | R9901 |

## 批次分組方式

`main.py` 的 `group_records_by_project_group` 依 **(專案, 群組)** 分組，同一 (專案, 群組) 的工號會合併成一個批次；順序為「檔案中首次出現的 (專案, 群組) 順序」。

上述兩行會變成 **2 個批次**（順序不變）：

1. **批次 1**：`(RL6483, Digital, [R9901])`
2. **批次 2**：`(RL6665, Digital, [R9901])`

## 主程式是否「每專案分次、等 3～5 分鐘再做下一個」？

**是，目前設計就是這樣運作。**

- **同一專案**：先依序處理該專案內所有群組（Digital、Analog、APR、DV 等），每個群組只做「Add {群組}..」→ 彈窗填工號、Submit → 等小視窗關閉，**不按儲存變更**。全部群組都加完後，**只按一次「儲存變更」**，再等 process 視窗關閉（約 3～5 分鐘）。
- **換專案時**：導向下一專案 UPDB 頁 → 變更成員名單 → 同樣先加完該專案所有群組，最後再按一次儲存變更。
- 因此會先做完 RL6483 的所有群組加人 → 儲存變更一次（3～5 分鐘）→ 再切到 RLE0655，加完 Digital / Analog / APR / DV 後 → 儲存變更一次。

## 總結

| 項目                   | 說明 |
|------------------------|------|
| 輸入                   | 兩行 → 2 筆 Record：(RL6483, Digital, R9901)、(RL6665, Digital, R9901) |
| 批次                   | 2 個批次，順序：RL6483 Digital → RL6665 Digital |
| 執行順序               | 先處理 RL6483（該專案所有群組加完 → 儲存變更一次、3～5 分鐘），完成後再處理下一專案 |
| 是否符合「每專案分次設定」 | 是，每個專案只按一次儲存變更，等該次跑完才進行下一個專案 |
